CAPITOLO 03

Estrazione dati con SPARQL

INTRODUZIONE TECNICA

Questa sezione documenta la prima fase del nostro lavoro, dedicata all'estrazione preliminare dei dati di Barry Lyndon tramite SPARQL. Prima di mappare le lacune, l'obiettivo è stato quello di analizzare l'architettura dei metadati esistenti su Wikidata, prendendo confidenza con l'endpoint risolvendo numerose sfida da noi proposte:

PREMESSA: I Namespace Fondamentali

L'interfaccia web di Wikidata Query Service ha una comoda funzione di autocompletamento che gestisce i prefissi comuni dietro le quinte. Tuttavia, per garantire il massimo rigore metodologico nella nostra analisi formale, abbiamo scelto di esplicitare sempre i namespace fondamentali all'inizio di ogni interrogazione.

  • PREFIX wd: <http://www.wikidata.org/entity/>
    Utilizzato per identificare le entità fisiche o concettuali (es. il film Barry Lyndon o il regista Stanley Kubrick).
  • PREFIX wdt: <http://www.wikidata.org/prop/direct/>
    Utilizzato per dichiarare le proprietà dirette che creano relazioni tra le entità (es. "regista", "data di pubblicazione").
  • PREFIX rdfs: <http://www.w3.org/2000/01/rdf-schema#>
    Il vocabolario base di RDF Schema, indispensabile per estrarre le label (etichette testuali) leggibili dall'uomo al posto dei codici alfanumerici.
QUERY 1

Identificazione del film e euristica REGEX

Il primo passo è l'identificazione univoca della risorsa. Abbiamo utilizzato una scansione sulle label testuali del grafo per trovare il codice identificativo wd:Q471716. 

SELECT DISTINCT ?film ?label
WHERE {
  ?film wdt:P31 wd:Q11424 ;           # È un film
        rdfs:label ?label .
  FILTER(REGEX(?label, "Barry Lyndon", "i")) 
} LIMIT 10

ANALISI DEI RISULTATI

La query restituisce un elenco di label in diverse lingue. quindi per restringere il campo dovrei usare un FILTER(LANG(?label) = "en"), ma la ricerca risulterebbe molto lenta in quanto il motore SPARQL in questo caso cercherebbe di scansionare miliardi di triple rdfs:label per trovare tutte quelle che contengono "Barry Lyndon" e poi controllare se sono in inglese, abbiamo quindi optato per un cambio di strategia per la ricerca.

QUERY 2

Ottimizzazione e dati di regia

Per velocizzare la ricerca, abbiamo sostituito la scansione REGEX con il legame diretto (BIND) all'identificativo trovato, estraendo contemporaneamente i metadati su regista e data di pubblicazione. 

SELECT DISTINCT ?film ?label ?registaLabel ?dataPubblicazione
WHERE {
  BIND(wd:Q471716 AS ?film) 
  ?film rdfs:label ?label . FILTER(LANG(?label) = "en")
  OPTIONAL {
    ?film wdt:P57 ?regista .
    ?regista rdfs:label ?registaLabel . FILTER(LANG(?registaLabel) = "en")
  }
  OPTIONAL { ?film wdt:P577 ?dataPubblicazione . }
} ORDER BY ?dataPubblicazione

I risultati appariranno qui...

ANALISI DEI RISULTATI

Questa query è istantanea. L'uso di OPTIONAL garantisce la restituzione del record anche in caso di mancanza di una specifica proprietà, mentre il filtro LANG pulisce il risultato dai duplicati linguistici. In questo caso la query ci restituiste un elenco di film duplicati in quanto la data di pubblicazione è multipla.

QUERY 3A

Mappatura dei Riconoscimenti tramite Blocco VALUES

In questo primo approccio, utilizziamo la clausola VALUES per assegnare in modo esplicito e inline le etichette testuali alle proprietà del grafo di Wikidata. Associamo rigidamente la proprietà dei premi ricevuti (wdt:P166) allo stato "Vinto" e la proprietà delle candidature (wdt:P1411) allo stato "Nomination". 

PREFIX wd: <http://www.wikidata.org/entity/>
PREFIX wdt: <http://www.wikidata.org/prop/direct/>
PREFIX rdfs: <http://www.w3.org/2000/01/rdf-schema#>

SELECT DISTINCT ?stato ?premioLabel
WHERE {
  # Assegnazione inline stabili di proprietà e stati
  VALUES (?proprieta ?stato) {
    (wdt:P166  "Vinto")
    (wdt:P1411 "Nomination")
  }

  # Interrogazione dei premi legati a Barry Lyndon (Q471716)
  wd:Q471716 ?proprieta ?premio .
  
  # Recupero dell'etichetta del premio
  ?premio rdfs:label ?premioLabel .
  FILTER(LANG(?premioLabel) = "en")
}
ORDER BY ?stato

I risultati appariranno qui...

ANALISI CRITICA DEI RISULTATI

Il Limite di VALUES: Sebbene questa query sia strutturalmente lineare, l'output evidenzia un problema logico: il sistema considera "Nomination" e "Vinto" come flussi ed entità del tutto distinte. In questo approccio un premio verrebbe considerato 2 volte se ha vinto, in quanto ha ricevuto anche una nomination.

QUERY 3B

Frazione dei Flussi tramite UNION Semplice

Per superare la rigidità dell'approccio precedente ed evitare la duplicazione forzata dei valori, abbiamo implementato l'operatore UNION. Questo costrutto permette di combinare i pattern di tripla relativi alle vittorie e alle candidature, isolando le ricerche all'interno di blocchi logici distinti. 

PREFIX wd: <http://www.wikidata.org/entity/>
PREFIX wdt: <http://www.wikidata.org/prop/direct/>
PREFIX rdfs: <http://www.w3.org/2000/01/rdf-schema#>

SELECT DISTINCT ?stato ?premioLabel
WHERE {
  {
    wd:Q471716 wdt:P166 ?premio . # P166: Premio ricevuto
  }
  UNION
  {
    wd:Q471716 wdt:P1411 ?premio . # P1411: Candidato a
  }
  
  ?premio rdfs:label ?premioLabel .
  FILTER(LANG(?premioLabel) = "en")
}
ORDER BY ?stato

I risultati appariranno qui...

ANALISI CRITICA DEI RISULTATI

L'Isolamento dello Scope: Eseguendo la query notiamo che la colonna dello stato risulta vuota. In SPARQL, ogni ramo di una UNION è un'unità completamente isolata. Se provassimo a scrivere un'istruzione BIND dentro le singole parentesi graffe per assegnare il testo "Vinto" o "Nomination", quella variabile nascerebbe e morirebbe solo all'interno di quel frammento. Quando la query esce dalla UNION per unire i flussi, perde i riferimenti locali impedendo la creazione di un'etichetta centralizzata.

QUERY 3C

Logica Condizionale Esterna con BOUND, IF e SAMPLE

La soluzione definitiva prevede il raddoppiamento delle variabili all'interno dei rami della UNION per marcare l'origine del dato tramite "variabili segnaposto" (?checkVinto e ?checkNomination). Spostando la logica condizionale all'esterno del blocco di unione, siamo in grado di mappare gli stati e aggregare i risultati. 

PREFIX wd: <http://www.wikidata.org/entity/>
PREFIX wdt: <http://www.wikidata.org/prop/direct/>
PREFIX rdfs: <http://www.w3.org/2000/01/rdf-schema#>

SELECT ?premioLabel (SAMPLE(?stato) AS ?statoUnivoco)
WHERE {
  {
    wd:Q471716 wdt:P166 ?premio . 
    wd:Q471716 wdt:P166 ?checkVinto . 
  }
  UNION
  {
    wd:Q471716 wdt:P1411 ?premio . 
    wd:Q471716 wdt:P1411 ?checkNomination . 
  }

  # Logica condizionale esterna allo scope isolato della UNION
  BIND(IF(BOUND(?checkVinto), "Vinto", 
       IF(BOUND(?checkNomination), "Nomination", "Sconosciuto")) AS ?stato)

  ?premio rdfs:label ?premioLabel .
  FILTER(LANG(?premioLabel) = "en")
}
GROUP BY ?premioLabel
ORDER BY ?statoUnivoco ?premioLabel

ANALISI CRITICA E METODOLOGICA

Questo costrutto rappresenta l'ottimizzazione ottimale del flusso semantico. Ecco perché le funzioni lavorano in modo sinergico:

  • Raddoppiamento delle variabili: Interroghiamo la proprietà due volte per ramo. La prima istanza mappa l'URI del premio (?premio, comune a tutta la query), la seconda istanza popola una variabile di controllo esclusiva del ramo. Se il film ha vinto un premio, si riempirà ?checkVinto; se è una candidatura, si riempirà ?checkNomination.
  • Funzione BOUND: È un operatore booleano esterno che verifica l'effettiva esistenza e valorizzazione di una variabile nel record.
  • Funzione IF: Se BOUND(?checkVinto) è vero, assegna la stringa personalizzata "Vinto". Se è falso, passa in cascata al controllo successivo. Questo permette di iniettare nel dataset finale etichette testuali pulite e leggibili che non esistono nativamente all'interno del Knowledge Graph.
  • Clausola GROUP BY: Raggruppa e unifica tutte le righe di dati che condividono la medesima stringa testuale del nome del premio (?premioLabel).
  • Funzione di aggregazione SAMPLE: Poiché Wikidata contiene spesso fonti multiple o triple duplicate per la medesima entità, il raggruppamento collasserebbe i dati generando errori visivi. SAMPLE ordina al motore SPARQL di estrarre un solo campione rappresentativo dello stato per quel gruppo di premi, azzerando i duplicati ridondanti nella tabella.
QUERY 4

Audit Globale delle Proprietà

L'ultima query esegue una scansione completa di tutte le proprietà associate a Barry Lyndon, filtrando gli identificativi esterni per concentrarsi solo sui dati semantici descrittivi. Inoltre con la clausola SAMPLE, forziamo l'estrazione di un solo esempio rappresentativo, in modo da evitare righe plurime per le proprietà. 

SELECT ?propLabel ?descrizioneProp (SAMPLE(?valoreLabel) AS ?esempio)
WHERE {
  wd:Q471716 ?p ?valore .
  ?prop wikibase:directClaim ?p .
  ?prop rdfs:label ?propLabel . FILTER(LANG(?propLabel) = "en")
  OPTIONAL { ?prop schema:description ?descrizioneProp . FILTER(LANG(?descrizioneProp) = "en") }
  OPTIONAL { ?valore rdfs:label ?valoreLabel . FILTER(LANG(?valoreLabel) = "en") }
  FILTER(!REGEX(?propLabel, "ID|identifier|identificativo", "i"))
} GROUP BY ?propLabel ?descrizioneProp ORDER BY ?propLabel
